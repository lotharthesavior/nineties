# Nineties Framework — Event Sourcing Refactor Plan

> Generated by a multi-agent council: Event Sourcing Architect · QA Engineer · Senior Dev (DX) · Sales Strategist · Technical Writer · Framework Architect · HIPAA Compliance Specialist
>
> Reference gold standard: `~/Code/terminal-docker/lambda-go-template` (Go, event sourcing + NATS JetStream)

---

## Context

The `nineties` Rust framework has an excellent foundation: `nineties-core` defines clean trait-based event sourcing (Aggregate, EventStore, CommandBus, EventBus, Projection, ProjectionEngine), and `nineties-es-sqlite` provides a working SQLite EventStore. **But none of it is wired to actual operations.** User mutations still go through direct Diesel calls. The InProcessEventBus has no distributed backend. Projections exist but are unused.

This plan bridges that gap — step by step, with small pointed changes — ending at full event sourcing + NATS JetStream parity, publishable reusable crates, and a HIPAA-capable framework.

**Positioning:** *The opinionated Rust framework where audit trails are a consequence of the architecture, not a feature you configure.*

---

## The 5 Steps

### Step 1 — Wire UserAggregate Through CommandBus

Build the missing `/api/v1/register` endpoint and corresponding `create_user` logic in `user_service.rs` to serve as the reference write path.

Replace any direct Diesel mutations in `api_controller.rs` with proper ES commands routed through the existing `CommandBus`. Create `UserAggregate` implementing the `Aggregate` trait. Every write goes through `handle() → EventStore::append()`. Establish naming conventions, Makefile targets, and workspace scaffolding. Write the "Adding Your First Aggregate" tutorial.

**Why first:** Every subsequent step publishes what the EventStore has. If writes bypass the EventStore after Step 3, the projection and JetStream are fed stale/wrong data — split-brain is baked in.

**Validation — Visual:** Run app + admin/event-log view. Create/update/delete user from UI or HTTP client. Confirm event log shows `UserRegistered`/`ProfileUpdated`/`UserDeleted` in strict sequence, and protected profile endpoint resolves from aggregate UUID in JWT.

**Validation — Automated:** Run Step 1 integration suite (`test_create_user_emits_user_registered_event`, duplicate conflict, update consistency, delete tombstone, command-bus-failure-no-event). Include assertion that write-path DB table (`users`) is no longer mutated directly by command handlers.

### Step 2 — Wire Projections + SQLite Read Models

Implement `SqliteReadModelStore` (real SQL via r2d2, not the in-memory stub). Create `UserProjector` building a `users_view` table from user events, idempotent (`INSERT OR REPLACE WHERE version < excluded`). Register with `ProjectionEngine` via `InProcessEventBus`. Replace GET controller Diesel reads with `users_view` queries. Add NATS JetStream to `docker-compose.yml` (infrastructure ready even if code doesn't use it yet).

**Why second:** JetStream consumers in Step 3 need idempotent projection handlers that are already proven correct in-process before adding distributed delivery. Replay of old events must be a no-op before the first durable consumer is created.

**Validation — Visual:** Create user then edit profile. Refresh profile/list pages and confirm UI reflects `users_view` changes without reading legacy `users` table. Reset projection, replay events, and confirm screens converge back to the same state.

**Validation — Automated:** Run `SqliteReadModelStore` unit tests (`execute/query/truncate`), projection snapshot integration tests, duplicate-event idempotency tests, and replay-from-zero test asserting deterministic final `users_view` state.

### Step 3 — Add NATS JetStream EventBus (`nineties-es-nats`)

Create `crates/nineties-es-nats`: implement `EventBus` using `async-nats` JetStream. `NatsEventBus::publish()` serializes each `Event` and publishes to `events.<aggregate_type>.<event_type>` (lowercase). Create durable consumer in `NatsEventBus::new()` (idempotent `get_or_create_stream`). Add `NATS_URL`, `NATS_STREAM` to config. Wire `NatsEventBus` into `CommandBus` construction alongside `InProcessEventBus`. Move `ProjectionEngine` to the worker process (Step 4).

**Why third:** The worker (Step 4) subscribes to JetStream. Without events publishing to JetStream here, the worker has nothing to consume. Stream must exist and events must flow before a durable consumer is created.

**Validation — Visual:** Open NATS monitor (`:8222`) and app/admin event stream side-by-side. Trigger user writes and confirm subjects under `events.user.*` receive messages while app still serves expected responses.

**Validation — Automated:** Run NATS container integration tests: publish/consume roundtrip, stream/consumer idempotent creation, at-least-once redelivery behavior, and per-aggregate ordering checks. Keep `EventStoreContract` suite green across stores.

### Step 4 — Add `nineties-worker` Crate (JetStream Projector)

Create `crates/nineties-worker`: standalone binary connecting to NATS with a durable consumer (`AckExplicit`, `DeliverAll`, 30s `ack_wait`) and driving `ProjectionEngine`. Per message: deserialize `Event` → `projection_engine.process(&event).await` → `msg.ack()` on Ok, `msg.nak()` on Err. Proves the full async event-driven projection pipeline before swapping the storage layer in Step 5.

**Why fourth:** All correctness properties (command validation, event ordering, optimistic concurrency, idempotent projection, JetStream delivery, durable consumption) must be verified against SQLite before touching the storage layer. Mixing a storage migration with consumer architecture introduces the hardest-to-diagnose replay bugs.

**Validation — Visual:** Run app + worker in separate terminals. Perform writes while tailing worker logs; verify each event transitions `received -> projected -> acked`. Stop worker, issue writes, restart worker, and confirm backlog replays to correct UI/read-model state.

**Validation — Automated:** Run worker lifecycle tests (happy path ACK, NAK+redelivery retry, graceful shutdown with in-flight message), plus concurrent dispatch stress test confirming no duplicate aggregate versions and no ordering violations.

### Step 5 — Production Storage, Reusable Crates, HIPAA Foundations

Create `crates/nineties-es-postgres`: `PostgresEventStore` via `sqlx::PgPool`, schema matching the Go gold standard (`UNIQUE(aggregate_id, version)`, `JSONB` payload). Create `nineties-rm-postgres`. Add `DATABASE_DRIVER=sqlite|postgres` config — swap `Box<dyn EventStore>` at startup with zero changes to domain code. Extract `nineties-core`, `nineties-es-sqlite`, `nineties-es-nats` as publishable crates. Add HIPAA compliance foundations (see HIPAA Appendix below). Write complete framework documentation.

**Validation — Visual:** Start app once with `DATABASE_DRIVER=sqlite` and once with `DATABASE_DRIVER=postgres`; execute identical user flow and compare output/event-log behavior. Demo projection rebuild and audit view to show parity and compliance fields present in UI/admin output.

**Validation — Automated:** Run full workspace tests under both drivers, per-crate independent CI build/test, `cargo publish --dry-run`, migration tests for Postgres schema constraints, and HIPAA audit assertions for required metadata (`user_id`, command type, timestamp, source info).

#### `crates/nineties-app/src/domain/user/aggregate.rs`
```rust
use crate::domain::user::commands::UserCommand;
use async_trait::async_trait;
use nineties_core::{aggregate::Aggregate, event::Event};
use thiserror::Error;

#[derive(Debug, Error)]
pub enum UserAggregateError {
    #[error("user already exists")]       AlreadyExists,
    #[error("user not found")]            NotFound,
    #[error("invalid email format")]      InvalidEmail,
}

#[derive(Default)]
pub struct UserAggregate {
    pub id:            Option<String>,
    pub name:          Option<String>,
    pub email:         Option<String>,
    pub password_hash: Option<String>,
    pub version:       i64,
    pub exists:        bool,
}

#[async_trait]
impl Aggregate for UserAggregate {
    type Command = UserCommand;
    type Error   = UserAggregateError;

    fn aggregate_type() -> &'static str { "User" }
    fn version(&self) -> i64 { self.version }

    async fn handle(&self, cmd: Self::Command) -> Result<Vec<Event>, Self::Error> {
        match cmd {
            UserCommand::RegisterUser { ref id, ref name, ref email, ref password_hash } => {
                if self.exists { return Err(UserAggregateError::AlreadyExists); }
                if !email.contains('@') { return Err(UserAggregateError::InvalidEmail); }
                Ok(vec![Event::new("User", id, self.version + 1, "UserRegistered",
                    serde_json::json!({ "id": id, "name": name,
                                        "email": email, "password_hash": password_hash }))])
            }
            UserCommand::UpdateProfile { ref id, ref name } => {
                if !self.exists { return Err(UserAggregateError::NotFound); }
                Ok(vec![Event::new("User", id, self.version + 1, "ProfileUpdated",
                    serde_json::json!({ "name": name }))])
            }
            UserCommand::ChangeEmail { ref id, ref email } => {
                if !self.exists { return Err(UserAggregateError::NotFound); }
                if !email.contains('@') { return Err(UserAggregateError::InvalidEmail); }
                Ok(vec![Event::new("User", id, self.version + 1, "EmailChanged",
                    serde_json::json!({ "email": email }))])
            }
            UserCommand::ChangePassword { ref id, ref password_hash } => {
                if !self.exists { return Err(UserAggregateError::NotFound); }
                Ok(vec![Event::new("User", id, self.version + 1, "PasswordChanged",
                    serde_json::json!({ "password_hash": password_hash }))])
            }
        }
    }

    fn apply(&mut self, event: &Event) {
        self.version = event.sequence;
        match event.event_type.as_str() {
            "UserRegistered" => {
                self.id            = Some(event.payload["id"].as_str().unwrap().to_string());
                self.name          = Some(event.payload["name"].as_str().unwrap().to_string());
                self.email         = Some(event.payload["email"].as_str().unwrap().to_string());
                self.password_hash = Some(event.payload["password_hash"].as_str().unwrap().to_string());
                self.exists        = true;
            }
            "ProfileUpdated"  => { self.name = Some(event.payload["name"].as_str().unwrap().to_string()); }
            "EmailChanged"    => { self.email = Some(event.payload["email"].as_str().unwrap().to_string()); }
            "PasswordChanged" => { self.password_hash = Some(event.payload["password_hash"].as_str().unwrap().to_string()); }
            _ => {}
        }
    }
}
```

### Critical Architectural Decisions for Step 1

**Decision 1: `CommandBus::dispatch` must take `&self`, not `&mut self`.**
The command bus has no mutable state — it delegates to `EventStore` and `EventBus`, which are `Arc`-backed. `&mut self` means you cannot share the bus across Actix request handlers without locking the entire bus per request. Fix: make `dispatch` take `&self`. Storage-layer locking (SQLite transactions via `spawn_blocking`) handles concurrency correctly.

**Decision 2: `CommandBusError` → HTTP status code mapping must be established now.**
Define `impl ResponseError for CommandBusError` in `nineties-app` (adapter layer, not core): `HandleFailed` → 422, `ConcurrencyConflict` → 409, `AggregateNotFound` → 404, everything else → 500. Do not leave this to each controller author — 10 engineers will produce 10 different behaviors.

**Decision 3: Actor identity flows into every event from Step 1 onward.**
Add `actor_id: Option<String>` to a `CommandContext` wrapper (or to the `Command` trait). `CommandBus::dispatch` stamps `actor_id` into every event's metadata automatically. Every future audit query depends on this being correct from the start. Retrofitting it later requires rewriting every event in the store.

## Framework Fit Assessment

**Best-suited production use cases:**
1. Healthcare / patient records (audit trail required, temporal queries on clinical history)
2. SaaS billing and subscriptions (complex lifecycle invariants, dispute resolution needs history)
3. Inventory and warehouse management (stock derived from stream of receipts/picks, oversell prevention)
4. Multi-tenant admin tools (server-rendered HTMX stack + real domain model)
5. Workflow / approval systems (state machines with full history)

**Top 3 production risks to address before shipping Step 1:**

1. **No snapshot support.** `CommandBus::dispatch` loads the full event stream every command. A 3-year-old user aggregate with 500 events loads 500 rows on every login. Add `save_snapshot`/`load_snapshot` to `EventStore` and `to_snapshot()`/`from_snapshot()` to `Aggregate` — interface in Step 1, implementation in Step 2.

2. **`InProcessEventBus::publish` blocks the write path.** A slow handler (email, Stripe, search index) blocks the entire HTTP response. If a handler panics, `CommandBus::dispatch` returns an error even though events were already persisted — write-side/read-side consistency violation. Fix: separate in-process synchronous handlers (fast, in-transaction) from async side-effects (JetStream, Step 3). Document this boundary clearly.

3. **`ReadModelStore` abstraction leaks SQL dialect.** The `execute(sql, params)` interface passes raw SQL strings treated as table names by `InMemoryReadModelStore`. Every projector is implicitly tied to one backend's SQL dialect. Redesign before projectors proliferate — either query-object based or explicitly parameterized.

**SQLite vs. Postgres decision:** SQLite is correct for single-process dev and embedded. Postgres is required the moment you add a second process, need `SKIP LOCKED`, or need `LISTEN/NOTIFY`. Document the boundary explicitly. Note: current `SqliteEventStore` casts `i64 sequence` and `u64 timestamp` to `i32` — silent data loss bug that must be fixed before any production use.

**NATS JetStream as optional, not required:** Keep `EventBus` for in-process synchronous delivery. Add a separate `DistributedEventBus` trait (ack/nack, consumer groups, durable subscriptions) that `NatsJetStreamBus` implements. Do not hide JetStream ack semantics behind the simple `EventBus` interface — it silently degrades when messages aren't acked.

## HIPAA Compliance Appendix

### Framework vs. Developer Responsibility

The framework provides technical safeguard *infrastructure*. It does not make applications automatically HIPAA-compliant. The `HIPAA_COMPLIANCE.md` file (Step 5) must state this clearly and include the developer checklist below.

**What the framework provides (documented as included):**

| Requirement | Component | §164 Citation |
|---|---|---|
| Audit metadata on every event | `AuditMetadata` struct in `EventStore` | §164.312(b) |
| PHI access logging infrastructure | `PhiAccessLogger` trait + stream | §164.312(b) |
| Session idle timeout enforcement | Configurable session middleware | §164.312(a)(2)(iii) |
| TLS required in production | HTTP binding layer | §164.312(e) |
| NATS mTLS enforcement | `NatsConfig` builder | §164.312(e) |
| Event payload encryption trait | `EventEncryption` trait + AES-256-GCM default | §164.312(a)(2)(iv) |

### Step 1 — HIPAA Tasks

These 5 tasks should land in Step 1 (interface defined now, full impl may extend to Step 2):

**HIPAA-1: Mandatory `AuditMetadata` on `EventStore` writes**
Define `AuditMetadata` in `nineties-core`: `actor_id`, `actor_session_id`, `source_ip`, `user_agent`, `timestamp_utc_us`, `event_sequence`, `causation_id`. Make `EventStore::append()` reject events without populated `AuditMetadata`. Breaking change — cheaper to make in Step 1 than retrofit later.

**HIPAA-2: Generic `AccessLogger` trait definition**
Define a generic `AccessLogger` trait in `nineties-core` with `log_access(actor: &Identity, resource: AccessedResource, purpose: PurposeOfUse)`. The resource carries a `Sensitivity` tag (`Phi`, `Pci`, `Pii`, `Confidential`, `Internal`, `Public`) so one mechanism satisfies HIPAA, GDPR, PCI-DSS, and SOC 2 — PHI is one regime among several. `NoOpAccessLogger` is the default for non-regulated apps and tests. JetStream sink lands in Step 3. Read events are required by §164.312(b) and equivalents; reads do not go through `EventStore`, so this is the only mechanism that captures them.

**HIPAA-3: Session idle timeout middleware**
Add `SessionConfig::idle_timeout` field (default: 900s for PHI apps per OCR guidance). Track `last_active_at` in session. Enforce independently from the existing 24h TTL. §164.312(a)(2)(iii) — Automatic Logoff.

**HIPAA-4: Server-side session store with token binding**
JWT `jti` (JWT ID) validated against server-side session store on every request. Store: `token_hash`, `user_id`, `created_at`, `last_active_at`, `revoked`. Enables immediate session revocation required for breach response. JWT-only auth is insufficient for PHI applications — token theft is undetectable until expiry. §164.312(d) — Person Authentication.

**HIPAA-5: HMAC event chain — interface and test harness**
Define `IntegrityChain` trait in `nineties-core`: `sign_event(event, prev_hash) -> EventSignature`, `verify_chain(events) -> IntegrityResult`. Implement HMAC-SHA256 chain with known test vectors. Wire into `EventStore` in Step 2. §164.312(c) — Integrity Controls.

### NATS JetStream HIPAA Risks

1. **Plaintext persistence by default** — application-layer encryption (AES-256-GCM) is mandatory before publishing PHI events. NATS disk storage never holds plaintext PHI. §164.312(a)(2)(iv).

2. **Subject namespace leakage** — subject names (e.g., `events.patient.12345.vitals`) reveal PHI in transit. Use opaque UUIDs in subjects, never natural keys. §164.312(e).

3. **Coarse-grained consumer access** — one consumer credential = access to all events in a stream. Design stream topology around data sensitivity, not just domain. Per-tenant or per-patient stream isolation via `StreamIsolationPolicy`. §164.312(a), §164.502(b).

4. **`PurgeStream` destroys all patients' data** — Right-to-erasure (GDPR/state law) must use cryptographic erasure (revoke DEK for that patient's events) not `PurgeStream`. `KeyManagement::revoke_key(key_id)` required. §164.524.

5. **Audit trail gaps on connectivity loss** — `PhiAccessLog` stream must use `AckExplicit`. HTTP response must not confirm until PHI access log event is acknowledged. No silent audit gaps. §164.312(b).

6. **Cluster peer communication is not mTLS by default** — NATS server config must include `cluster { tls { ... } }`. Framework deployment docs must require this explicitly. §164.312(e)(1).
